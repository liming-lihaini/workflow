## 1. 项目概述

### 1.1 项目背景

随着企业数字化转型的深入，企业内部管理系统面临着日益复杂的业务流程管理需求。传统的流程审批往往依赖硬编码的审批逻辑，缺乏灵活性和可扩展性，导致业务流程变更成本高、响应慢。因此，企业需要一套可自定义、可扩展的流程引擎，来支撑多级审批、业务流程编排、任务分配与协作等多种业务场景。

### 1.2 项目目标

本项目旨在构建一套**可自定义的流程引擎**，提供以下核心能力：

1. **可视化流程设计**：支持通过JSON配置定义流程，支持前端可视化拖拽设计
2. **自定义节点类型**：通过插件机制，支持灵活扩展节点实现复杂业务逻辑
3. **表单与数据收集**：支持在流程节点中收集和处理用户输入数据
4. **独立部署运行**：作为独立服务，不依赖现有系统，可快速集成

### 1.3 适用范围

- 企业内部审批流（请假、报销、采购等）
- 业务流程编排（订单处理、供应链、业务办理等）
- 任务分配与协作（项目任务、多人协作处理）

## 2. 功能需求

### 2.1 核心功能模块

流程引擎由以下核心模块组成：

| 模块 | 功能描述 |
|------|---------|
| 流程设计器 | 定义流程模板、节点、连线、触发条件 |
| 流程运行引擎 | 解析流程定义、执行节点逻辑、处理流转 |
| 任务中心 | 管理待办任务、已办任务、任务认领与转交 |
| 表单引擎 | 定义节点表单、收集用户输入、数据校验 |
| 节点插件体系 | 提供可扩展的节点类型实现 |
| 流程监控 | 流程实例监控、历史记录、异常处理 |
| 后台管理 | 部门管理、用户管理、角色管理、权限管理、日志管理 |

### 2.2 流程设计器

#### 2.2.1 流程模板管理

- 支持创建、编辑、删除流程模板
- 支持流程模板的复制和版本管理
- 支持流程模板的导入导出（JSON格式）

#### 2.2.2 流程设计

- 支持通过JSON配置定义流程结构
- 支持定义以下流程元素：
  - **节点（Node）**：流程中的步骤单元
  - **连线（Edge）**：节点之间的流转关系
  - **网关（Gateway）**：条件分支、并行处理
  - **事件（Event）**：开始、结束、触发器

#### 2.2.3 流程设计JSON结构

```json
{
  "id": "leave_request",
  "name": "请假申请流程",
  "version": "1.0",
  "nodes": [
    {
      "id": "start",
      "type": "start",
      "name": "开始",
      "position": {"x": 100, "y": 100}
    },
    {
      "id": "apply",
      "type": "userTask",
      "name": "提交申请",
      "formKey": "leave_form",
      "position": {"x": 250, "y": 100},
      "assignee": "${initiator}"
    },
    {
      "id": "approve_level1",
      "type": "userTask",
      "name": "一级审批",
      "formKey": "approve_form",
      "position": {"x": 400, "y": 100},
      "assignee": "${manager}"
    },
    {
      "id": "gateway_approve",
      "type": "exclusiveGateway",
      "name": "审批结果判断",
      "position": {"x": 550, "y": 100}
    },
    {
      "id": "end",
      "type": "end",
      "name": "结束",
      "position": {"x": 700, "y": 100}
    }
  ],
  "edges": [
    {"id": "e1", "source": "start", "target": "apply"},
    {"id": "e2", "source": "apply", "target": "approve_level1"},
    {"id": "e3", "source": "approve_level1", "target": "gateway_approve"},
    {"id": "e4", "source": "gateway_approve", "target": "end", "condition": "${approved == true}"},
    {"id": "e5", "source": "gateway_approve", "target": "apply", "condition": "${approved == false}"}
  ]
}
```

### 2.3 流程运行引擎

#### 2.3.1 流程实例管理

- 支持发起流程实例
- 支持查询流程实例列表和详情
- 支持流程实例的暂停、恢复、终止操作
- 支持流程实例变量（Context）管理

#### 2.3.2 节点执行机制

- **顺序执行**：按连线顺序依次执行节点
- **条件执行**：根据节点条件判断下一个节点
- **并行执行**：支持并行网关，多个分支同时执行
- **回退执行**：支持流程回退到指定节点

#### 2.3.3 流程变量

- 支持在流程级别定义变量（Process Variables）
- 支持在节点级别定义变量（Node Variables）
- 支持使用EL表达式访问变量
- 支持变量作用域：全局、流程实例、节点

#### 2.3.4 流程时效管理

- **流程级时效**
  - 支持设定流程整体处理时限（如：7天内完成）
  - 支持设定流程催办阈值（如：超过5天未完成触发催办）
  - 支持设定提醒频率（如：每天催办一次）

- **节点级时效**
  - 支持设定单个节点处理时限（如：审批节点3天内完成）
  - 支持设定节点催办阈值（如：超过2天未处理触发催办）
  - 支持设定节点提醒频率
  - 支持工作日和自然日两种计算方式

- **时效计算**
  - 自动记录流程已停留时长
  - 自动记录节点已停留时长
  - 自动计算超时时长
  - 支持按时效维度统计流程处理效率

- **超时处理**
  - 支持超时后自动通过
  - 支持超时后自动驳回
  - 支持超时后自动转办
  - 支持超时后触发回调通知

- **自动提醒**
  - 支持到达催办阈值自动提醒
  - 支持超时后自动提醒
  - 支持多种提醒方式（站内消息、邮件、短信）
  - 支持按提醒频率重复提醒
  - 支持提醒通知到处理人、负责人、申请人

- **时效配置示例**
  ```json
  {
    "id": "approve",
    "type": "userTask",
    "name": "部门审批",
    "timeout": {
      "duration": 172800,        // 48小时（秒）
      "unit": "workday",         // workday-工作日, natural-自然日
      "reminderThreshold": 86400, // 24小时前开始催办
      "reminderFrequency": 43200, // 每12小时催办一次
      "timeoutAction": "autoTransfer", // timeoutAction: none-无操作, autoPass-自动通过, autoReject-自动驳回, autoTransfer-自动转办
      "reminderMethods": ["站内消息", "邮件"]
    }
  }
  ```

### 2.4 任务中心

#### 2.4.1 任务管理

- 支持任务签收（Claim）和取消签收（Unclaim）

### 2.5 特殊审批模式

#### 2.5.1 会签模式（Counter Sign）

会签是指多个审批人同时收到任务，需要全部审批通过或按指定规则通过后才能流转到下一节点。

- **会签类型**：
  - **全通过**：所有审批人全部同意，流程才能通过
  - **按比例**：按比例通过即可（如50%以上同意）
  - **一票通过**：任一审批人同意即可通过
  - **一票否决**：任一审批人反对则流程驳回

- **会签配置**：
  ```json
  {
    "id": "approve_group",
    "type": "userTask",
    "name": "部门会签",
    "signType": "counterSign",
    "signMode": "allPass",  // allPass, ratio, onePass, oneReject
    "signRatio": 0.5,
    "assignees": ["${deptMembers}"],
    "counterSignRule": {
      "passCount": 3,
      "rejectCount": 1,
      "timeout": 86400
    }
  }
  ```

- **会签特性**：
  - 所有会签人可以看到彼此的审批意见
  - 支持会签结果汇总展示
  - 支持会签意见的汇总导出
  - 支持会签超时自动通过/驳回
  - 支持会签节点完成后自动知会申请人

#### 2.5.2 加签模式（Add Sign）

加签是指在审批过程中，原审批人临时增加其他审批人，原审批人继续有效。

- **加签类型**：
  - **前加签**：新增审批人在原审批人之前处理
  - **后加签**：新增审批人在原审批人之后处理
  - **并行加签**：新增审批人与原审批人并行处理

- **加签配置**：
  ```json
  {
    "id": "approve_manager",
    "type": "userTask",
    "name": "经理审批",
    "allowAddSign": true,
    "addSignTypes": ["before", "after", "parallel"],
    "assignee": "${manager}"
  }
  ```

- **加签特性**：
  - 支持在审批过程中随时发起加签
  - 支持指定加签人和加签顺序
  - 支持加签意见的回传
  - 支持加签转交给其他人
  - 支持加签完成后返回原审批人
  - 支持加签历史记录查看

#### 2.5.3 审批操作扩展

- **审批动作**：
  - 同意（Approve）
  - 驳回（Reject）
  - 加签（Add Sign）
  - 转办（Transfer）
  - 委派（Delegate）
  - 撤回（Withdraw）
  - 取消（Cancel）

- **审批意见**：
  - 支持填写审批意见
  - 支持上传审批附件
  - 支持选择常用意见模板

#### 2.5.4 审批流程示意

```
会签模式示例（全部通过）：
┌─────────┐    ┌─────────────────┐    ┌─────────┐
│ 提交申请 │───▶│  部门会签(3人)   │───▶│  结束   │
└─────────┘    │  [审批人A] ✓    │    └─────────┘
               │  [审批人B] ✓    │
               │  [审批人C] ✓    │───▶ 流程通过
               └─────────────────┘

                     │                  │
                     └──────────────────┴──▶ 流程继续
```

#### 2.6.3 表单数据绑定

- 支持表单数据的持久化存储

#### 2.6.4 流程表单操作权限

流程节点绑定表单后，需要配置表单在流程中的操作权限，包括节点级别权限和字段级别权限。

##### 2.6.4.1 节点级别表单权限

节点级别控制表单的整体操作能力：

```json
{
  "id": "apply",
  "type": "userTask",
  "name": "提交申请",
  "formKey": "leave_form",
  "formPermission": {
    "mode": "edit",
    "allowSaveDraft": true,
    "allowWithdraw": true,
    "allowCancel": false,
    "allowPrint": true,
    "readonly": false
  }
}
```

| 属性 | 说明 | 可选值 |
|------|------|-------|
| mode | 表单模式 | edit（可编辑）、readonly（只读）、hidden（隐藏） |
| allowSaveDraft | 允许保存草稿 | true/false |
| allowWithdraw | 允许撤回 | true/false |
| allowCancel | 允许取消流程 | true/false |
| allowPrint | 允许打印 | true/false |
| readonly | 是否只读 | true/false |

##### 2.6.4.2 字段级别操作权限

字段级别控制每个表单字段的操作能力：

```json
{
  "id": "approve",
  "type": "userTask",
  "name": "审批",
  "formKey": "leave_form",
  "fieldPermissions": [
    {
      "field": "leaveType",
      "permission": "readonly"
    },
    {
      "field": "startDate",
      "permission": "hidden"
    },
    {
      "field": "reason",
      "permission": "edit",
      "required": false
    },
    {
      "field": "approveComment",
      "permission": "edit",
      "required": true
    }
  ]
}
```

| 权限值 | 说明 |
|--------|------|
| edit | 可编辑 |
| readonly | 只读（显示但不可修改） |
| hidden | 隐藏（不显示） |
| required | 必填（仅edit模式生效） |

##### 2.6.4.3 字段权限继承与覆盖

支持按层级配置字段权限，子节点可以继承或覆盖父节点的配置：

```json
{
  "id": "approve_level1",
  "type": "userTask",
  "name": "一级审批",
  "formKey": "leave_form",
  "fieldPermissions": [
    {
      "field": "reason",
      "permission": "readonly"
    }
  ],
  "permissionInherit": {
    "enabled": true,
    "inheritFrom": "apply"
  }
}
```

##### 2.6.4.4 条件化字段权限

支持根据流程变量或业务条件动态设置字段权限：

```json
{
  "id": "approve",
  "type": "userTask",
  "name": "审批",
  "formKey": "leave_form",
  "fieldPermissions": [
    {
      "field": "urgent",
      "permission": "edit",
      "condition": "${isUrgent == true}"
    },
    {
      "field": "managerComment",
      "permission": "readonly",
      "condition": "${currentUser.role != 'manager'}"
    }
  ]
}
```

##### 2.6.4.5 按钮级别操作权限

控制表单提交按钮的显示和操作：


### 2.7 节点插件体系

#### 2.7.1 内置节点类型

| 节点类型 | 说明 |
|---------|------|
| start | 开始节点，流程的入口 |
| end | 结束节点，流程的出口 |
| userTask | 用户任务节点，需要人工处理 |
| serviceTask | 服务任务节点，自动执行逻辑 |
| exclusiveGateway | 排他网关，根据条件选择分支 |
| parallelGateway | 并行网关，并行执行多个分支 |
| inclusiveGateway | 包容网关，根据条件执行多个分支 |
| scriptTask | 脚本任务节点，执行脚本逻辑 |
| subProcess | 子流程节点，调用子流程 |

#### 2.7.2 自定义节点扩展

- 提供节点接口（NodeHandler），支持自定义实现
- 支持节点级别的生命周期钩子：
  - `onEnter`：进入节点前
  - `onExecute`：执行节点逻辑
  - `onLeave`：离开节点后
- 支持节点配置自定义属性
- 支持节点国际化

#### 2.7.3 节点接口定义

```java
public interface NodeHandler {
    
    /**
     * 节点类型标识
     */
    String getNodeType();
    
    /**
     * 进入节点
     */
    default void onEnter(ExecutionContext context) {
        // 可重写
    }
    
    /**
     * 执行节点逻辑
     */
    void execute(ExecutionContext context);
    
    /**
     * 离开节点
     */
    default void onLeave(ExecutionContext context) {
        // 可重写
    }
    
    /**
     * 获取节点配置Schema
     */
    default Map<String, Object> getConfigSchema() {
        return Collections.emptyMap();
    }
}
```

### 2.8 流程监控

#### 2.8.1 流程实例监控

- 支持查看流程实例当前状态
- 支持查看流程执行轨迹（节点执行顺序）

#### 2.9.9 三员管理规范

遵循安全保密要求的三员管理规范，实现权限分离：

- **系统管理员**
  - 负责系统基础配置
  - 负责用户账号管理（除安全员、审计员外的普通用户）
  - 负责系统参数配置
  - 权限范围：系统配置、用户管理、基础数据维护

- **安全管理员**
  - 负责安全策略配置
  - 负责角色和权限分配（除系统管理员、审计员外的用户权限）
  - 负责安全参数配置（密码策略、登录策略等）
  - 权限范围：角色管理、权限分配、安全策略、密级管理

- **审计管理员**
  - 负责日志查看和导出
  - 负责日志审计分析
  - 负责三员操作日志审计
  - 权限范围：访问日志查看、操作日志查看、日志导出

- **三员权限约束**
  - 三员之间不能互相操作
  - 三员不能删除自己的账号
  - 三员操作行为需记录审计日志
  - 系统初始化时创建三个管理员账号

### 4.3 数据模型建模

为了支持复杂业务场景的表单数据管理，流程引擎提供**数据模型**功能，允许用户定义数据结构，并在表单设计中绑定数据模型。

#### 4.3.1 数据模型概述

- **数据模型（DataModel）**：定义业务数据的结构模板，包含字段定义、字段类型、校验规则等
- **模型实例（ModelInstance）**：数据模型的一次具体使用，存储实际的业务数据
- **主子表关系**：一个数据模型可以包含多个子表，形成层级结构，支持一对多关系

#### 4.3.2 数据模型定义

数据模型支持以下特性：

| 特性 | 说明 |
|------|------|
| 字段类型 | 支持文本、数字、金额、日期、文件、人员、部门等 |
| 字段校验 | 支持必填、长度、正则表达式、自定义函数校验 |
| 主子表 | 支持定义主表和多个子表，形成层级结构 |
| 关联引用 | 支持跨模型引用，关联其他数据模型 |
| 计算字段 | 支持配置计算公式，自动计算字段值 |

#### 4.3.4 表单绑定数据模型

表单设计时，可以直接绑定数据模型，实现表单字段与数据模型字段的自动映射。

##### 绑定方式

| 绑定类型 | 说明 |
|---------|------|
| 模型绑定 | 选择完整的数据模型，自动生成表单所有字段 |
| 字段绑定 | 选择数据模型中的单个或多个字段 |
| 子表绑定 | 绑定子表字段，支持动态增删行 |

##### 表单绑定配置示例

```json
{
  "formKey": "expense_form",
  "formName": "报销申请表单",
  "modelKey": "expense_model",
  "layout": "grid",
  "sections": [
    {
      "sectionKey": "basic",
      "sectionName": "基本信息",
      "table": "main",
      "fields": ["expenseNo", "title", "applicant", "department", "expenseDate"]
    },
    {
      "sectionKey": "details",
      "sectionName": "费用明细",
      "table": "detail",
      "fields": ["expenseType", "description", "amount", "invoiceNo", "attachment"],
      "editable": true,
      "addable": true,
      "deletable": true
    },
    {
      "sectionKey": "approvers",
      "sectionName": "审批配置",
      "table": "approvers",
      "fields": ["level", "approver", "role"],
      "editable": true,
      "addable": true,
      "deletable": true
    }
  ]
}
```

##### 字段映射关系

| 表单字段 | 数据模型字段 | 映射类型 |
|---------|------------|---------|
| expenseNo | main.expenseNo | 自动生成 |
| title | main.title | 双向绑定 |
| applicant | main.applicant | 双向绑定 |
| detail[].expenseType | detail.expenseType | 双向绑定 |
| detail[].amount | detail.amount | 双向绑定 |
| totalAmount | main.totalAmount | 计算得出 |

#### 4.3.6 数据模型与流程集成

数据模型可以与流程引擎深度集成：

1. **流程发起时创建模型实例**：发起流程时自动创建数据模型实例
2. **流程变量传递模型数据**：表单数据可传递到流程变量
3. **节点读取模型数据**：服务节点可读取模型数据进行业务处理
4. **流程结束时归档模型实例**：流程结束时自动归档模型实例数据

##### 流程定义中引用数据模型

```json
{
  "id": "expense_process",
  "name": "报销申请流程",
  "modelKey": "expense_model",
  "nodes": [
    {
      "id": "apply",
      "type": "userTask",
      "name": "提交报销",
      "formKey": "expense_form",
      "dataBinding": {
        "modelKey": "expense_model",
        "instanceIdVar": "expenseInstanceId",
        "onComplete": "save"
      }
    },
    {
      "id": "approve",
      "type": "userTask",
      "name": "审批",
      "formKey": "approve_form",
      "dataBinding": {
        "modelKey": "expense_model",
        "instanceId": "${expenseInstanceId}",
        "mode": "readonly"
      }
    }
  ]
}
```

### 4.4 数据模型与表单引擎的关系

数据模型是对表单引擎的增强和补充，两者关系如下：

| 特性 | 表单引擎 | 数据模型 |
|------|---------|---------|
| 数据结构 | 单表结构 | 主子表结构 |
| 数据校验 | 字段级校验 | 字段级+行级校验 |
| 数据关联 | 无 | 支持跨模型关联 |
| 计算能力 | 无 | 支持计算字段 |
| 数据持久化 | 依赖流程实例 | 独立管理 |

- **轻量场景**：使用传统表单引擎即可满足需求
- **复杂业务**：推荐使用数据模型，支持主子表和计算字段

## 8. 验收标准

### 8.1 功能验收

- [ ] 可以通过JSON创建流程定义
- [ ] 可以发起流程实例并正常流转
- [ ] 可以在用户任务节点完成审批
- [ ] 支持条件分支流转
- [ ] 支持并行网关
- [ ] 可以定义表单并收集数据
- [ ] 可以开发自定义节点并正常执行
- [ ] 任务中心可以正常查询和管理任务

#### 8.1.1 会签功能验收

- [ ] 支持配置会签节点（全通过/按比例/一票通过/一票否决模式）
- [ ] 会签任务可以同时分配给多个审批人
- [ ] 可以查看所有会签人的审批意见
- [ ] 会签结果可以按配置规则自动汇总
- [ ] 支持会签超时自动处理

#### 8.1.2 加签功能验收

- [ ] 支持在审批过程中发起加签（前加签/后加签/并行加签）
- [ ] 可以指定加签人和加签顺序
- [ ] 加签完成后可以返回原审批人
- [ ] 可以查看加签历史记录

#### 8.1.3 表单操作权限验收

- [ ] 支持节点级别表单权限配置（edit/readonly/hidden模式）
- [ ] 支持字段级别操作权限配置（edit/readonly/hidden）
- [ ] 支持配置字段是否必填
- [ ] 支持配置按钮级别操作权限（提交/保存草稿/驳回）
- [ ] 支持权限继承与覆盖机制
- [ ] 支持条件化字段权限（根据流程变量动态控制）
- [ ] 表单绑定数据模型时支持子表增删改权限控制
- [ ] 字段权限在流程节点间正确传递和应用

### 8.2 性能验收

- [ ] 流程实例创建响应时间 < 500ms
- [ ] 节点执行响应时间 < 200ms
- [ ] 支持1000并发流程实例

### 8.2 数据模型验收

- [ ] 可以创建数据模型，定义主表和子表结构
- [ ] 支持配置字段类型、必填校验、自定义校验

- [ ] 支持计算字段，自动汇总子表数据
- [ ] 数据模型实例可以独立创建、查询、更新
- [ ] 数据模型可以与流程集成，流程发起时自动创建模型实例

### 8.3 稳定性验收

- [ ] 连续运行7天无异常
- [ ] 支持集群部署
- [ ] 数据可以正常持久化
