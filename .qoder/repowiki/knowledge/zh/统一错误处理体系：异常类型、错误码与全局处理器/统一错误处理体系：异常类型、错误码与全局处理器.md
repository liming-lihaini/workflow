## 错误处理架构概览

该项目采用 Spring Boot 标准的全局异常处理机制，构建了统一的错误响应体系。核心设计围绕三个关键组件：统一错误码枚举、业务异常类、全局异常处理器。

## 核心组件

### 1. 统一错误码体系 (`ErrorCode`)
- 位于 `com.flow.engine.common.ErrorCode`
- 采用枚举定义，遵循 HTTP 状态码约定：0 成功、4xx 客户端错误、5xx 系统错误、1xxx 业务错误
- 涵盖流程引擎、权限管理、数据模型、Webhook 等所有业务域的错误场景
- 每个错误码包含 code 和 message 两个字段

### 2. 异常类型分层
- **BusinessException**: 通用业务异常，继承 RuntimeException，默认使用 BUSINESS_ERROR 错误码
- **FlowException**: 流程引擎专用异常，提供更灵活的构造方式，支持自定义错误码
- 两者都实现了 IResultCode 接口规范

### 3. 全局异常处理器 (`GlobalExceptionHandler`)
- 使用 `@RestControllerAdvice` 注解实现 AOP 式异常拦截
- 统一处理四类异常：
  - BusinessException → 返回 200 + Result.fail()
  - MethodArgumentNotValidException → 参数校验失败，返回 400
  - NoHandlerFoundException → 路由不存在，返回 404
  - Exception → 兜底处理，返回 500 系统错误

### 4. 统一响应结构 (`Result`)
- 所有 API 响应统一封装为 Result<T> 对象
- 包含 code、message、data、requestId、timestamp 字段
- 提供 ok()、error()、fail() 等便捷工厂方法
- 自动注入 requestId 用于链路追踪

## 使用模式

### 业务层抛异常
```java
throw new BusinessException(ErrorCode.PROCESS_NOT_DEPLOYED, "流程定义未部署");
throw new FlowException(1001, "表达式求值失败");
```

### 工具层异常包装
```java
try {
    // 业务逻辑
} catch (FlowException e) {
    throw e; // 直接抛出业务异常
} catch (Exception e) {
    throw new FlowException(1001, "操作失败: " + e.getMessage());
}
```

### 控制器层直接返回
```java
return Result.ok(data);
return Result.error(ErrorCode.PARAM_INVALID, "参数错误");
```

## 设计决策

1. **异常与响应分离**：业务层抛出异常，由全局处理器转换为统一响应格式
2. **错误码集中管理**：所有错误码在 ErrorCode 枚举中统一定义，避免魔法数字
3. **分层异常设计**：通用业务异常与领域特定异常分离，便于不同层次处理
4. **HTTP 语义保留**：虽然业务异常返回 200，但系统异常仍保持正确的 HTTP 状态码
5. **请求追踪集成**：Result 自动携带 requestId，便于问题排查

## 开发规范

1. 业务异常优先使用 BusinessException + ErrorCode 组合
2. 框架内部异常使用 FlowException
3. 不要在业务代码中捕获并吞掉异常，应向上抛出
4. 工具方法应将底层异常包装为 FlowException 抛出
5. 新增错误码时应在 ErrorCode 枚举中统一定义
6. 前端应根据 code 而非 HTTP 状态码判断业务结果
